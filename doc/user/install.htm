Install Documentation
Welcome

The GeoServer project is a Java implementation of the 1.0 Web Feature Server specification from the OpenGIS Consortium.   GeoServer aspires to be the 'Apache' of OpenGIS data serving and its mission is to enable greater geographic interoperability by enforcing OpenGIS standards and lowering the barriers to entry for geographic data providers.

Before you can use this GeoServer beta, you must successfully install Java, a Java Servlet container such as Resin or Tomcat, and PostGIS. This is by far the hardest part of the installation process, but the documentation to install these three components is quite good. Our testing has occurred on JDK 1.4.1, Tomcat 4.0.4, and PostGIS 0.7.3 on top of Postgresql 7.2.2. While installing PostGIS make sure to include the JDBC drivers in the PostgreSQL compile.

GeoServer has an active user community that you can engage by joining the GeoServer mailing list and asking questions or making suggestions.  If you create a user account at this site, you may participate in forums and vote in polls as well.
Installing Java
GeoServer is built against the JDK1.4.1.  If you do not have the JRE1.4.1 installed on your machine, you should download it from Sun and point to it in your JAVA_HOME path.
Installing a Servlet Container
Several good open source, free servlet containers and a bevy of commercial ones exist on the market.  We recommend Resin (open source, free to non-commercial users) or Tomcat (open source, free).  Most GeoServer testing has occured on Resin, which is remarkably simple to set up and administer.  We reccomend using the stand-alone server installation over the integrated Apache install.  Given the maturity of these projects, we will say nothing further about installation.
Installing a Database

Although it is a primary goal of the GeoServer project to support several data storage formats, good support exists now only for PostGIS, an excellent extension of the venerable Postgresql relational database.  PostGIS and Postgresql are open source databases that are free to all users. Postgresql was developed at Berkeley several years ago and PostGIS is a recent extension developed by the clever programmers at Refractions Research. Installing Postgresql and PostGIS is the most complicated part of setting up a GeoServer site and for this reason, we will supplement the documentation to do so a little here.

First, you should download the full source of Postgresql.  Binaries, RPMs, and other packaged builds will generally not suffice because you will need to compile the PostGIS extension and JDBC bindings into Postgresql in order for it to correctly speak with GeoServer.

First, download Postgresql - we have had the best luck with version 7.2.3.  You should configure and install it, per the installation instructions.  It should go something like this:

./configure --with-java
gmake
gmake install

Make sure that you configure it with Java.   Be sure that it is up and running, it's best to create a database and test it using psql from the command line.  Then, download PostGIS.  You should place PostGIS in your Postgresql 'contrib' directory, change to the postgresql root directory and follow the directions for installation there.

You should, of course, check to make sure that Postgresql and PostGIS are functioning correctly before you attempt to connect with then using GeoServer.  After installing postgis attempt to create tables with geometry columns, and insert a few features.
Installing GeoServer

If you have an existing GeoServer installation on your machine, you should backup your geoserver/data/featureTypes directory before placing the new .war file. To create the .war file you must run ant war from the geoserver root directory. This creates the geoserver.war in the same directory.

Move 'geoserver.war' from the GeoServer root directory to the war file directory of your Java Servlet container (i.e. 'webapps' in Resin). You may run a simple test by by starting your container and entering this address:

http://SERVER_ADDRESS:SERVER_PORT/geoserver/GetCapabilities

Note that SERVER_ADDRESS and SERVER_PORT should be replaced by your server's address and port. If GeoServer is running, you should see an XML capabilities document. If you get nothing, then you should consult your server documentation to see what has gone wrong. If a geoserver directory has not been created automatically by your server, then your server is probably not behaving as it should.
Configuring GeoServer

This section assumes at least passing familiarity with XML files.  However, if you simply follow the test file example exactly, you should be able to create a working configuration file.  A sample configuration.xml file resides at geoserver/misc/documents/ You may edit this file, and changes will be picked up when you run ant war Or you may go to your root web application (webapps in tomcat or resin) directory and edit the following file: geoserver/data/configuration.xml.  This sets the global configuration options for your server and is defined by a schema that exists in the root geoserver directory ast the file: misc/schema/configuration.xsd.  An example configuration.xml is shown here:

<?xml version="1.0" encoding="UTF-8" ?>
<GlobalConfiguration version="0.0.13" updateSequence="0">
    <Name>GeoServer</Name>
    <Title>The VFNY OWS1 Basemap Server</Title>
    <Abstract>This is a test server. It contains some basemap data from New York City.</Abstract>
    <Keywords>WFS, OGS, OWS1</Keywords>
    <OnlineResource>http://beta.vfny.org/geoserver</OnlineResource>
    <Fees>This is a free server.</Fees>
    <AccessConstraints>There are no access constraints to this server.</AccessConstraints>
    <URL>http://beta.vfny.org/geoserver</URL>
    <Maintainer>Vision for New York</Maintainer>
</GlobalConfiguration>

You should modify the values between the tags only.  All values are defined below:

Option Name Description

Name

 This is the name of your server and is arbitrary.
Title This is the title of the server and is arbitrary
Abstract This is a brief description of what your server does and is arbitrary.
Keywords A series of comma seperated keywords, describing the data and purpose of the server.
OnlineResource A unique url where information for this server resides. Even if you have no real online source it is important to put something here, for default namespace surfaces. It is fine to use the same url as the URL element.
URL Must be the exact base URL of the server or client connections will fail.

Maintainer

 The name of the maintainer for this server.

If you would like to correctly service 0.0.15 capabilities, you must also modify the WSDL document in the geoserver/data/capabilities folder.  It is beyond the scope of this document to explain these modifications, but they are not particularly complex.

For more configuration options, check the Advanced section.
Adding Data to GeoServer

Next, you should go to your root web application directory and navigate to the following directory: data/featureTypes/.  For every table in your PostGIS database, you should create a directory which matches the table name exactly.

This directory must contain two files: info.xml and schema.xml.  The schema.xml document is the feature type schema and its structure is defined in the GML 2.1 specification.  The info.xml is specific to GeoServer and must be correctly specified in order for you to serve a given feature type.  Here is an example info.xml file for a feature tupe named rail:

<?xml version="1.0" encoding="UTF-8" ?>
<featureType xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
    <Name>rail</Name>
    <Title>NYC Rail Centerlines</Title>
    <Abstract>This is a simple rail coverage from the New York City basemap.</Abstract>
    <Keywords>OWS1, rail, railroad, World Trade Center, New York City</Keywords>
    <SRS>32118</SRS>
    <LatLonBoundingBox minx="-74.27000" miny="40.50000" maxx="-73.80000" maxy="40.94000" />
    <DatasourceParams>
       <host>localhost</host>
       <port>5432</port>
        <database>test</database>
       <user>test</user>
       <passwd>test</passwd>
       <table>road</table>
       <dbtype>postgis</dbtype>
    </DatasourceParams>
</featureType>

The info.xml file is used by GeoServer and its fields are a superset of those returned per feature type by the GetCapabilities document defined by the WFS specification.  There are also additional database fileds.  The SRS is currently the most important field and its value should correspond to an approved EPSG code only (no GML namespace, which is added by the server intenally.

Those experienced with GeoServer may notice a few changes in this file. This is part of an effort to provide different datasources with minimal user configuration. The DatasourceParams tag is now used, with the appropriate datasource fields. The old tags will still work for postgis, but future additional datasources will only support the new style. The metadata tags have been removed since they were not used, though they may come back in a future version of geoserver. The operations tag was never really used, and now will be determined by the datasource's advertised capabilities.

Option Name Description

Name

 This must match the PostGIS table name exactly. Yes, this is repeated in tablename, but for now both are required.
Title This is the title of the feature type and is arbitrary
Abstract This is a brief description of what your feature type represents and is arbitrary.
Keywords A series of comma seperated keywords, describing the data in this feature type of the server.
LatLonBoundingBox Must have attributes in geographic coordinates denoting feature type bounding box.
host Must match the PostGIS postmaster daemon URI (serving feature type) exactly, port excluded.
port Must match the PostGIS postmaster daemon port (serving feature type) exactly.
database Must match the PostGIS database name (holding feature type) exactly.
user Must match the PostGIS database user (with access to feature type) exactly.
passwd Must match the PostGIS database user password (with access to feature type) exactly.
dbtype Must always be postgis, as it is the only currently supported db.
table Must match the PostGIS table name exactly.

The schema.xml file must also be correctly specified for GeoServer to work correctly.  To create a schema.xml file, you must be capable of creating valid XML schemas (in general) and valid GML schemas (in particular).  This remains a pretty obscure skill, so we have written a small tutorial on it here.  The 'topp:road' directory is included as an example, but should be deleted before you do any serving (unless you also have a table called 'road' in the topp namespace).

 
Obtaining test data
The easiest way to get some postgis data to work with is to acquire shapefiles and use the shp2pgsql utility provided by postgis. It should be in your pgsql installation directory, under the bin subdirectory. As for acquiring shapefiles, the US census provides a lot of their data free of cost in the shape format, available here. The latest release of geoserver also includes support for using shapefiles directly, information is about this option is available in the advanced section. Using the shapefiles directly sacrifices a lot of speed and flexibility, so converting them to postgis is the recommended way, but the option is available for easier set up.
Testing GeoServer
This release includes an extremely simple WFS testing utility, written in Java. It uses XML files to specify a variety of inputs such as host, request, and service.  You may use this utility to determine whether or not you have correctly installed and configured GeoServer.  The operation of this utility is described in greater depth here.  Happy GeoServing!
