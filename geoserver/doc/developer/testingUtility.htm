Testing Utility Documentation
Introduction

The WFS Tester is an open source, command-line Java utility that will pass structured requests to a WFS, receive a response, and make minimal assessments of the response body.  It could be described as the world's stupidest WFS client.  It is useful to those who would like to run either structured regression or ad-hoc tests on installed web feature servers.  If you want to get and use the WFS Tester, you must download it seperately from the GeoServer download page.
Running the Utility

The WFS Tester is found in the wfsTester/libraries directory and is labeled wfsTester.jar.  To run the utility, you should type:

java -cp $GEOSERVER_ROOT/libraries/wfsTester.jar org.vfny.wfsTest.utilities.InterfaceTest [flags...]

where $GEOSERVER_ROOT is the root directory of your geoserver installation
and possible [flags...] are:

Flag Options Description
-f [root test data dir] [host file] [test specification file] interface test
-v   verbose output
--help   help

Example:

java -cp org.vfny.wfsTest.utilities.InterfaceTest -vf /home/rob/wfs/wfsTest/misc/data geoserver.xml test1.xml
Specifying Hosts

The host specification files may be found in dataDirectory/hosts, where dataDirectory is the user-specified data directory.

Attribute Description
hostName A descriptive host name.
ip The request URI.
port The host port; may leave empty.
directory The subdirectory under which responses occur; this value is geoserver for all GeoServer installations.

Example:

<testHost xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
    <hostName>VFNY GeoServer (external)</hostName>
    <ip>beta.vfny.org</ip>
    <port>81</port>
    <directory>geoserver</directory>
</testHost>
Specifying Tests

The test specification files may be found in dataDirectory/tests, where dataDirectory is the user-specified data directory.

Attribute Description
testTitle A descriptive title for the test; displayed while running.
requestInterface The request interface to be tested: GetCapabilities, DescribeFeatureType, or GetFeature.
requestFile Name of the text file containing the request (KVP or XML), found in dataDirectory/requests.
targetSchemaFile Name of the schema file with which to validate the response, found in dataDirectory/schemas.
targetOutputFile Name of the text file containing exact target output, found in dataDirectory/output.
isPost Value indicating whether or not the request is POST or GET; may be 'true' or 'false'.

Example:

<testSpecification xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" >
    <testTitle>Test GetFeature (subway)</testTitle>
    <requestInterface>GetFeature</requestInterface>
    <requestFile>getFeature4.txt</requestFile>
    <targetSchemaFile />
    <targetOutputFile />
    <isPost>true</isPost>
</testSpecification>

 
Conclusion

Keep in mind that this is only a helper utility and should not in any way be considered a definitive indication of specification compliance.  There are plans to further develop the parsing capabilities of this tool.  If you would like to contribute to its development, its source code is availible on CVS.
